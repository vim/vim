Wayland Protocol Support 				*wayland*

1. What is Wayland		|wayland-what|
2. Connecting to Wayland	|wayland-connection|
3. GUI support                  |wayland-gui|
4. Wayland selections		|wayland-selections|

==============================================================================
1. What	is Wayland					*wayland-what*

The Wayland protocol specifies a set of protocols that governs how the display
server, commonly known as a compositor, and clients communicate.  As a
protocol, the de facto library currently that implements it is libwayland, which 
Vim uses.  While Wayland clients are commonly associated to have windows, this
is not mandatory, and currently Vim only uses the data control protocols using
the Wayland protocol, see *wayland-selections* for more information.

==============================================================================
2. Connecting to Wayland				*wayland-connection*

For Vim to use the wayland compositor, it must be able to connect to one.  This
should generally not be a problem, as your desktop environment will likely set
the $WAYLAND_DISPLAY environment variable for you.  This environment variable
can be an absolute path to the wayland display file, which is generally a UNIX
domain socket.  Additionally, if it is relative (no leading slash), then it is
relative to the $XDG_RUNTIME_DIR environment variable value.  Additionally, you
can use the `:wlrestore` command to override the $WAYLAND_DISPLAY environment
variable.  If that doesn't work, then you can try falling back to X11, such as
if you have XWayland running, using the `:restoreclip` command.

==============================================================================
3. GUI support						*wayland-gui*

See |gui-wayland|.  Please note that when using the GUI, Vim uses the toolkit
such as GTK for accessing the clipboard, and does not access the clipboard
using libwayland.  You can check this though the |v:clipmethod| variable, which
should equal to "none" when running the GUI.

==============================================================================
4. Wayland Selections					*wayland-selections*

Vim currently supports the wlr-data-control-unstable-v1 and
ext-data-control-v1 protocols, for accessing the current Wayland selection.
Selection in this case essentially means the "clipboard."  You can check if
your Wayland compositor supports either of these protocols by running the
wayland-info command, which should be bunded with libwayland on your system: >
	wayland-info | grep -E '(ext_data_control|zwlr_data_control)'
<If grep finds a match, then you have either or both protocols on your system.
If you don't get any match, then please see *wayland-no-data-control* for more
information.

If you come from X11, then the regular wayland selection is equivalent to the
CLIPBOARD selection in X11, and the primary wayland selection equates to the
X11 primary selection.  Accessing these selections is the same as X11 in Vim,
in which the + register is the regular selection, and the * register is the
primary selection, note that your compositor may not support primary
selections, see *wayland-primary-selection* for more details.

							*wayland-and-x11*
If your version of vim comes compiled with both X11 and Wayland support, then
Vim determines which one to use when accessing the clipboard using the
'clipmethod' option.

						 *wayland-primary-selection*
If you find X11 style primary selections useful, then to check if your
compositor supports the primary-selection-unstable-v1 protocol, check if
zwp_primary_selection_device_manager_v1 is found using the method found in
*wayland-selections*, using wayland-info.  If your compositor does not support 
the protocol, then any interactions with the * register will use the regular 
selection instead.

				 *wayland-no-data-control* *wayland-gnome*
If you are using the GNOME desktop environment on Wayland, as of this writing,
there is no method of accessing/modifiying the clipboard for external clients
such as Vim without being focused.  Focused in this case means the client has
received some sort of input event, such as a window being focused.  This is
what the wlr-data-control-unstable-v1 and ext-data-control-v1 protocols
solve.  If your Wayland compositor does not support the above protocols, then
the above explanation applies.

Wayland specific commands:
							*:wlrestore* *:wl*
:wl[estore] [display]	Reinitializes the connection to the wayland compositor.
			[display] should be in the format of the $DISPLAY 
			environment variable (e.g. "wayland-0"), see
			*wayland-connection* for more details.  If [display]
			is omitted, then it reinitializes the connection using
			the same value as was used for the previous execution
			of this command.  If the value was never specified,
			then it uses the value of $WAYLAND_DISPLAY environment
			variable.
			{only available when compiled with the |+wayland| feature}

Wayland error messages:
							*E1546*
Vim failed communicating with the wayland compositor.  This is likely due to
the Wayland compositor process being killed.  Try the `:wlrestore` command to
try connecting again.  See *wayland-connection* for more details.

							*E1547*
Vim failed communicating with the wayland compositor.  See *E1546* for more
details.
